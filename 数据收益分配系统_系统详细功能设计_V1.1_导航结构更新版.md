# 1. 文档定位与设计原则

## 1.1 设计目标

本文档不是重复需求规格说明书，而是把需求转换为后续开发可用的功能设计输入。输出重点包括：

1.  明确每个业务模块在系统中的职责边界。
2.  明确页面、按钮、表单、状态、异常、日志和输出之间的关系。
3.  为数据库设计提供候选表、字段、关系、状态、枚举和约束输入。
4.  为 UI 设计提供页面路由、信息区块、按钮状态、弹窗、空状态、错误提示和 trace 展示要求。
5.  为接口设计提供服务分层、核心操作、输入输出和副作用。
6.  为测试设计提供模块级和按钮级验收映射。

## 1.2 设计边界

P0 版本以“本地可跑通、可演示、可导出、可审计”为边界，不要求登录、生产级 RBAC、PDF、异步队列、多租户和真实财务结算。P1 在 P0 稳定后扩展登录、权限、PDF、CSV/XLSX 导入、异步任务进度和历史报告管理。

系统输出为收益分配模拟参考，不构成法律结算、财务付款、合同履约或主管单位审批结果。所有页面与报告必须保留“模拟参考，非法律结算”说明。

## 1.3 功能设计原则

1.  **计算链路可分段执行**：数据接入、质量评估、数元计量、贡献效用、MD-DShap、收益分配、报告导出均可独立触发，也可由首页一键执行。
2.  **历史结果不可静默覆盖**：重新评估、重新计量、重新计算和重新导出均生成新版本。
3.  **算法结果只进入权重层**：MD-DShap 输出权重，不直接形成付款指令。
4.  **非数据主体默认不进算法池**：运营方、中试基地、技术服务方和专家方按合同优先或约束处理，默认不进入 MD-DShap。
5.  **导出字段固定化**：P0 产生 Markdown、CSV、JSON、JSONL；每个文件有明确字段清单。
6.  **审计快照贯穿全流程**：计算类操作必须保存输入快照、参数快照、输出快照和算法版本。
7.  **UI 先表达流程状态，再展示计算明细**：页面必须让用户知道当前处于哪个业务状态、哪些按钮可用、哪些前置条件未满足。

## 3.3 主业务链路

主链路按以下顺序推进：

1.  创建项目或选择演示数据。
2.  上传或初始化数据包，生成输入快照。
3.  识别数据资源、字段、模态和基础统计。
4.  维护参与方，区分数据源主体和非数据贡献主体。
5.  将数据资源关联到数据源主体。
6.  配置质量指标权重并执行质量评估。
7.  配置数元基准价、系数和调用量，执行数元计量。
8.  配置贡献因子，计算贡献度和归一化贡献。
9.  配置效用函数，计算效用值并保留 trace。
10. 以 MD-DShap 为默认算法计算参与方权重。
11. 配置总收益、合同优先分配和数据源收益池。
12. 选择分配模式并执行收益分配模拟。
13. 应用合同约束，生成约束前后金额。
14. 锁定分配方案或复制新版本重新计算。
15. 导出报告、明细、算法审计说明和审计日志。

# 4. 角色与权限功能设计

## 4.1 P0 与 P1 权限策略

P0 使用“本地操作员”模式，不强制登录，但每个操作仍需要在设计上保留操作人字段，默认值为 `local_operator`。P1 启用登录和 RBAC 后，用户、角色、权限、菜单和按钮需要进入统一权限模型。

## 4.2 权限动作类型

| **动作类型**   | **说明**             | **典型按钮**                                              |
|----------------|----------------------|-----------------------------------------------------------|
| VIEW           | 只读查看。           | 查看详情、预览、查看日志。                                |
| CREATE         | 新增数据。           | 新增参与方、新增约束、选择演示数据。                      |
| UPDATE         | 修改配置或业务对象。 | 编辑参与方、配置权重、配置参数。                          |
| DELETE_DISABLE | 删除、停用或撤回。   | 停用参与方、删除数据包、停用约束。                        |
| CALCULATE      | 触发计算。           | 启动质量评估、执行数元计量、启动 MD-DShap、执行收益分配。 |
| EXPORT         | 文件导出。           | 导出 CSV、生成 Markdown、导出审计日志。                   |
| CONFIRM        | 方案锁定或人工确认。 | 锁定分配方案、确认报告。                                  |

## 4.3 角色功能矩阵

| **角色**      | **允许重点操作**                               | **禁止重点操作**                 |
|---------------|------------------------------------------------|----------------------------------|
| 本地操作员    | P0 全流程操作、计算、导出、审计查看。          | 不代表生产级权限模型。           |
| 业务管理员    | 数据接入、参与方、收益池、分配模拟、报告导出。 | 不应独立修改系统级权限。         |
| 算法审核员    | 质量评估、效用计算、MD-DShap、算法审计导出。   | 不应维护合同优先金额。           |
| 合同审核员    | 合同约束、合同优先分配、分配结果审核。         | 不应修改算法参数。               |
| 审计员        | 只读查看报告、日志、快照、导出审计文件。       | 不应触发计算或修改参数。         |
| 普通查看用户  | 查看已确认结果和已发布报告。                   | 不应查看未确认草稿和计算 trace。 |
| 系统管理员 P1 | 用户、角色、权限、系统参数、模板。             | 不应替代业务确认。               |

## 5.2 全局页面布局规则

1.  顶部显示系统名称、当前项目、当前状态、当前操作人和风险提示入口。
2.  左侧菜单必须按照“系统首页、数据管理、数元贡献度计量、收益分配计算、报告生成与导出、系统管理”的顺序展示；系统首页不设二级菜单，数据管理下设数据接入管理、数据资源管理、参与方管理，数元贡献度计量下设质量评估管理、数元计量管理、贡献度与效用计算，收益分配计算下设 MD-DShap 计算管理、收益分配模拟、合同约束管理，系统管理下设参数配置、用户与权限管理（P1）、审计日志管理。
3.  页面顶部固定显示当前项目状态条：草稿、已接入、已评估、已计量、已计算效用、已计算权重、已分配、已确认、已导出。
4.  每个计算页面必须显示前置条件检查列表，未满足项以禁用按钮和明确文案提示。
5.  每个导出页面必须显示导出字段清单和“模拟参考，非法律结算”说明。
6.  每个查看 trace 的页面采用“摘要表 + 展开抽屉/弹窗”的交互，避免主表过宽。
7.  已锁定或已导出的项目，核心配置按钮默认置灰，用户只能复制新版本后修改。
8.  空状态必须说明下一步操作，而不是只显示“暂无数据”。
9.  错误提示必须定位到字段、前置条件或业务规则。
10. 所有重要操作包括删除、停用、重新计算、锁定方案、导出报告必须二次确认。

## 6.2 主外键关系建议

1.  `allocation_project` 为项目根对象，几乎所有业务表均应带 `project_id`。
2.  `data_package` 归属 `allocation_project`，一个项目可以存在多个数据包版本。
3.  `data_resource` 归属 `data_package`，并通过 `data_resource_party_relation` 关联 `party`。
4.  `party` 归属 `allocation_project`，项目内名称唯一。
5.  `quality_assessment` 引用 `data_package` 和质量参数快照。
6.  `shuyuan_metering` 引用 `quality_assessment`，明细引用 `data_resource` 和 `party`。
7.  `contribution_record` 与 `utility_record` 均以 `party_id` 为主体维度。
8.  `md_dshap_task` 引用效用版本和参数快照，`md_dshap_result` 和 `md_dshap_marginal_trace` 引用 `task_id`。
9.  `allocation_scenario` 引用项目、收益池配置、算法权重结果和合同约束版本。
10. `allocation_result` 引用 `allocation_scenario` 和 `party`，是分配导出的主事实表。
11. `report_record` 引用项目和输出快照，保存文件路径、校验摘要和版本。
12. `audit_log` 可引用任意对象，应保留 `object_type + object_id` 的通用字段。

# 8. MD-DShap 专题功能设计

## 8.1 产品定位

MD-DShap 是系统默认权重计算策略。基础 Shapley 仅在参与方数量较少时作为基线校验，不作为默认最终分配模式。MD-DShap 模块必须同时满足产品需求、算法需求和审计需求。

## 8.2 算法输入设计

| **输入项**       | **字段建议**                   | **来源模块**      | **校验规则**                                                         |
|------------------|--------------------------------|-------------------|----------------------------------------------------------------------|
| 数据源主体集合 P | participant_set                | 参与方管理        | 只允许 include_in_md_dshap=true 且 party_type=DATA_PROVIDER 的主体。 |
| 任务集合 T       | task_set                       | 数据资源/场景配置 | P0 可使用 default_task；多任务模式必须显式传入。                     |
| 效用函数 v(S,t)  | utility_function_snapshot_id   | 效用计算          | 必须有来源说明、公式说明和参数快照。                                 |
| 质量评分 q_i     | quality_factor / quality_score | 质量评估          | 取最新或用户选择的质量评估版本。                                     |
| 贡献度 c_i       | normalized_contribution        | 贡献度计算        | 合计应为 1。                                                         |
| 场景参数 s       | scenario_factor / scenario_id  | 系统参数/场景配置 | 系数必须大于 0。                                                     |
| 随机种子 seed    | seed                           | 算法参数          | 大于等于 0，默认 42。                                                |
| 采样轮次 n       | sample_rounds                  | 算法参数          | P1 完整开放；P0 可内置默认值。                                       |
| 收敛阈值 epsilon | epsilon                        | 算法参数          | 大于 0；P1 用于异步或长任务。                                        |

| **输出项**                  | **字段建议**                           | **用途**                             |
|-----------------------------|----------------------------------------|--------------------------------------|
| participant_weight          | participant_weight / normalized_weight | 参与方综合权重，进入收益分配。       |
| task_level_weight           | task_level_weight_json                 | 多任务下每个任务维度的权重。         |
| marginal_contribution_trace | md_dshap_marginal_trace                | 审计算法边际贡献过程。               |
| approximation_note          | approximation_note                     | 披露采样、动态估值、近似假设和边界。 |
| algorithm_audit_snapshot    | algorithm_audit_snapshot_id            | 报告导出和审计追溯。                 |

## 8.4 v(S,t) 与效用函数规则

P0 默认规则：

`v(S,t) = sum(utility_value_i,t for i in S) + synergy_adjustment(S,t)`

其中 `synergy_adjustment(S,t)` P0 默认可为 0，P1 可扩展为多模态协同、数据互补性、场景协同项或专家调整项。空集合规则为 `v(∅,t)=0`。单参与方场景不启动完整 MD-DShap，直接输出权重 1，并在页面和报告中披露“单主体简化分配”。

## 12.2 快照策略

1.  输入快照：上传 JSON、演示数据初始化、复制新版本时生成。
2.  参数快照：质量权重、数元系数、贡献因子、效用函数、MD-DShap 参数、收益池、合同约束在计算前生成。
3.  输出快照：每次计算成功后生成，导出报告引用输出快照。
4.  报告快照：每次导出报告时保存报告内容摘要、文件路径、checksum。
5.  历史快照不得修改；如需修订，生成新版本并保留来源版本号。

# 13. 数据库设计前置约束

1.  所有主业务表必须包含 `project_id` 或可通过外键追溯到 `project_id`。
2.  所有可重复执行的计算结果必须包含 `version_no`、`created_at`、`created_by`。
3.  金额字段使用 Decimal，不使用 float。
4.  权重字段统一保留 6 位，金额字段统一保留 2 位。
5.  历史结果引用的数据不得物理删除；删除类操作优先使用 status 标记。
6.  已导出报告引用的输入、参数、结果和约束必须不可变。
7.  大字段如输入快照、输出快照、边际贡献 trace 可使用 JSON 存储，但必须有 checksum。
8.  日志表应支持按项目、模块、对象、操作类型和时间范围查询。
9.  P1 用户权限表应支持菜单级、页面级、按钮级和导出级权限。
10. 报告文件路径和 checksum 必须入库，避免无法追溯导出版本。

# 14. UI 设计前置约束

1.  左侧导航必须采用六个一级菜单：系统首页、数据管理、数元贡献度计量、收益分配计算、报告生成与导出、系统管理；各二级功能必须与第 5.1 节一致。
2.  首页必须展示完整流程状态、项目总览、流程入口、风险提示和一键计算入口，避免用户不知道下一步做什么。
3.  每个模块顶部必须显示当前项目名称、状态、最近版本和风险提示。
4.  计算按钮必须先展示前置条件检查结果，未满足时禁用。
5.  表格操作列的高风险按钮必须二次确认。
6.  “重新计算”按钮必须提示不会覆盖历史结果。
7.  “导出”弹窗必须展示导出文件、字段范围、报告版本和免责声明。
8.  MD-DShap 页面必须突出“默认算法为 MD-DShap，基础 Shapley 仅作基线”。
9.  收益分配页面必须展示“合同优先分配 -\> 数据源收益池 -\> 权重分配 -\> 合同约束调整”的顺序。
10. 约束前金额和约束后金额必须并列展示，不能只展示最终金额。
11. 审计页面必须支持从日志跳转到输入快照、参数快照和输出快照。
12. 已确认/已导出项目的编辑按钮必须置灰，并提供“复制新版本”入口。
13. P1 登录后，权限不足的按钮应隐藏或禁用，并显示角色原因。

# 18. 结论

本文档在 V1.2 需求规格说明书基础上，将系统需求转换为功能模块、页面路由、按钮行为、候选表、服务边界、导出字段、审计快照和验收映射。本次导航结构更新后，左侧导航固定为“系统首页、数据管理、数元贡献度计量、收益分配计算、报告生成与导出、系统管理”六个一级菜单，并将原有功能模块映射到对应二级功能。后续数据库设计可直接依据第 6、11、12、13 章建立表结构和约束；UI 设计可直接依据第 5、7、8、14 章建立页面原型和交互流程；后端接口设计可依据第 10 章拆分服务和统一输入输出。

系统核心设计结论是：以数据接入为起点，以质量评估、数元计量和效用计算形成算法输入，以 MD-DShap 作为默认权重计算策略，以合同优先分配和合同约束调整形成最终模拟结果，以报告和审计日志完成输出追溯。系统结果仅为模拟参考，非法律结算。
