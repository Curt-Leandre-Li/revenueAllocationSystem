### D. 合同与收益分配真实链路

```text
保存合同比例方案
-> 校验 ratio_sum = 1.000000
-> 完成 MD-DShap 数据源主体权重
-> POST /projects/{project_id}/allocation/simulate
-> 非数据主体金额 amount_source=CONTRACT_RATIO
-> 数据源主体金额 amount_source=MD_DSHAP_WEIGHT
-> 尾差写入 allocation_result.rounding_delta
-> 锁定或导出报告
```

项目级合同比例路径不生成普通 `constraint_apply_trace`。旧 `contract_constraint` 和 `constraint_apply_trace` 只保留给旧 `allocation/simulation/run` 或 `allocation-scenarios` 兼容路径，不作为当前合同分配规则页主流程。

# 1. 文档定位与设计原则

## 1.1 设计目标

本文档不重复需求规格说明书，重点把需求拆成后续开发可用的页面、按钮、状态和服务设计。内容包括：

1.  明确每个业务模块在系统中的职责边界。
2.  明确页面、按钮、表单、状态、异常、日志和输出之间的关系。
3.  为数据库设计提供候选表、字段、关系、状态、枚举和约束输入。
4.  为 UI 设计提供页面路由、信息区块、按钮状态、弹窗、空状态、错误提示和 trace 展示要求。
5.  为接口设计提供服务分层、核心操作、输入输出和副作用。
6.  为测试设计提供模块级和按钮级验收映射。

## 1.2 设计边界

P0 版本以“本地可跑通、可演示、可导出、可审计”为边界，不要求登录、生产级 RBAC、PDF、异步队列、多租户和真实财务结算。P1 在 P0 稳定后扩展登录、权限、PDF、CSV/XLSX 导入、异步任务进度和历史报告管理。

系统输出为收益分配模拟参考，不构成法律结算、财务付款、合同履约或主管单位审批结果。所有页面与报告必须保留“模拟参考，非法律结算”说明。

## 1.3 功能设计原则

1.  计算链路可分段执行：数据接入、质量评估、数元计量、贡献效用、MD-DShap、收益分配、报告导出均可独立触发，也可由首页一键执行。
2.  历史结果不可静默覆盖：重新评估、重新计量、重新计算和重新导出均生成新版本。
3.  算法结果只进入权重层：MD-DShap 输出权重，不直接形成付款指令。
4.  非数据主体默认不进算法池：运营方、中试基地、技术服务方和专家方按合同比例分配规则处理，默认不进入 MD-DShap。
5.  导出字段固定：P0 产生 Markdown、CSV、JSON、JSONL；每个文件有明确字段清单。
6.  审计快照覆盖计算过程：计算类操作必须保存输入快照、参数快照、输出快照和算法版本。
7.  UI 先表达流程状态，再展示计算明细：页面必须让用户知道当前处于哪个业务状态、哪些按钮可用、哪些前置条件未满足。

## 3.3 主业务链路

主链路按以下顺序推进：

1.  创建项目或选择演示数据。
2.  上传或初始化数据包，生成输入快照。
3.  识别数据资源、字段、模态和基础统计。
4.  维护参与方，区分数据源主体和非数据贡献主体。
5.  将数据资源关联到数据源主体。
6.  配置质量指标权重并执行质量评估。
7.  配置数元基准价、系数和调用量，执行数元计量。
8.  配置贡献因子，计算贡献度和归一化贡献。
9.  配置效用函数，计算效用值并保留 trace。
10. 配置总收益。
11. 配置数据源收益池比例和非数据主体合同比例。
12. 按合同比例方案形成非数据主体合同金额与数据源收益池。
13. 以 MD-DShap 为默认算法计算数据源主体归一化权重。
14. 使用 MD-DShap 归一化权重分配数据源主体收益池。
15. 执行尾差处理，生成最终模拟金额和金额来源说明。
16. 锁定分配方案或复制新版本重新计算。
17. 导出报告、明细、算法审计说明和审计日志。

# 4. 角色与权限功能设计

## 5.2 全局页面布局规则

1.  顶部显示系统名称、当前项目、当前状态、当前操作人和风险提示入口。
2.  左侧菜单必须按照“系统首页、数据管理、数元贡献度计量、收益分配计算、报告生成与导出、系统管理”的顺序展示；系统首页不设二级菜单，数据管理下设数据接入管理、数据资源管理、参与方管理，数元贡献度计量下设质量评估管理、数元计量管理、贡献度与效用计算，收益分配计算下设 MD-DShap 计算管理、合同分配规则、收益分配模拟，系统管理下设参数配置、用户与权限管理（P1）、审计日志管理。
3.  页面顶部固定显示当前项目状态条：草稿、已接入、已评估、已计量、已计算效用、已计算权重、已分配、已确认、已导出。
4.  每个计算页面必须显示前置条件检查列表，未满足项以禁用按钮和明确文案提示。
5.  每个导出页面必须显示导出字段清单和“模拟参考，非法律结算”说明。
6.  每个查看 trace 的页面采用“摘要表 + 展开抽屉/弹窗”的交互，避免主表过宽。
7.  已锁定或已导出的项目，核心配置按钮默认置灰，用户只能复制新版本后修改。
8.  空状态必须说明下一步操作，而不是只显示“暂无数据”。
9.  错误提示必须定位到字段、前置条件或业务规则。
10. 所有重要操作包括删除、停用、重新计算、锁定方案、导出报告必须二次确认。

## 6.2 主外键关系建议

1.  `allocation_project` 为项目根对象，几乎所有业务表均应带 `project_id`。
2.  `data_package` 归属 `allocation_project`，一个项目可以存在多个数据包版本。
3.  `data_resource` 归属 `data_package`，并通过 `data_resource_party_relation` 关联 `party`。
4.  `party` 归属 `allocation_project`，项目内名称唯一。
5.  `quality_assessment` 引用 `data_package` 和质量参数快照，作为包级质量评估汇总。
5a. `quality_resource_assessment` 引用 `quality_assessment` 和 `data_resource`，保存逐资源质量评估主结果；`quality_resource_score_detail` 引用 `quality_resource_assessment`，保存资源级指标明细。
6.  `shuyuan_metering` 引用 `quality_assessment`，明细引用 `data_resource` 和 `party`。
7.  `contribution_record` 与 `utility_record` 均以 `party_id` 为主体维度。
8.  `md_dshap_task` 引用效用版本和参数快照，`md_dshap_result` 和 `md_dshap_marginal_trace` 引用 `task_id`。
9.  `allocation_scenario` 引用项目、收益池配置和算法权重结果；项目级合同比例方案在当前实现中通过 JSON runtime 对象关联，旧合同约束版本仅用于兼容路径。
10. `allocation_result` 引用 `allocation_scenario` 和 `party`，是分配导出的主事实表。
11. `report_record` 引用项目和输出快照，保存文件路径、校验摘要和版本。
12. `audit_log` 可引用任意对象，应保留 `object_type + object_id` 的通用字段。

## 6.3 字段类型与精度规则

| **字段类型** | **建议类型**         | **规则**                                      |
|--------------|----------------------|-----------------------------------------------|

| 权重         | Decimal(18,6)        | 权重合计需校验为 1，允许 0.000001 内误差。    |
| 分数         | Decimal(8,4)         | 质量分、贡献分、效用值可保留 4 位或 6 位。    |
| 系数         | Decimal(12,6)        | 所有系数必须大于 0，调用量除外。              |
| 调用量       | Integer / BigInteger | 必须大于等于 0。                              |
| 快照内容     | JSON / Text          | 需要 checksum，避免历史篡改。                 |
| 日志内容     | JSON / JSONL         | before/after 推荐 JSON。                      |
| 枚举         | String + 字典表可选  | P0 可硬编码，P1 可迁移到系统字典。            |

### 7.5.1 后端逐资源质量评估板块（新增）

设计定位：在不新增左侧导航的前提下，将“质量评估管理（QUAL）”的后端执行粒度扩展为资源级。页面仍从 `/metering/quality` 进入，`QUAL-003 启动质量评估` 触发后端编排；后端新增 `QUAL-010` 至 `QUAL-013` 作为服务级功能单元和可测试按钮/动作。

数据对象：QualityAssessment、QualityScoreDetail、QualityResourceAssessment、QualityResourceScoreDetail、DataResource、SnapshotStore、AuditLog。

当前 P0 页面展示：`/metering/quality` 顶部展示包级质量总分、等级、已评估资源数、平均资源评分和低分资源数；中部展示数据资源评分总览和资源 × 一级指标热力图；摘要图可展示前 8 条资源、热力图可展示前 6 条资源，但页面必须明确标注展示范围，并提供“查看全部资源评分”抽屉。资源表“查看评分”打开单资源指标详情，展示该资源 7 个一级指标和 17 个二级指标的 `score`、`weight`、`weighted_score`、`evidence_text`、`rule_code`、`assessment_id`、`resource_id`。

处理链路：

1.  读取 `project_id`、`package_id`、质量指标模板和参数快照。
2.  查询数据包下 `include_in_calculation=true` 且 `status=ACTIVE` 的 `data_resource`。
3.  为每个资源按模态选择规则集，执行 7 个一级指标和 17 个二级指标评分。
4.  写入 `quality_resource_assessment` 与 `quality_resource_score_detail`。
5.  返回包级 `package_score/package_level`、资源级 `average_resource_score/low_score_resource_count`、资源明细和热力图矩阵。
6.  生成结果快照和审计日志；部分失败时记录资源级失败详情。

| **功能编号** | **按钮/功能** | **业务行为** | **数据副作用与设计要点** |
|--------------|---------------|--------------|--------------------------|

资源级评分聚合规则：

`resource_score = Σ(first_level_weight × Σ(second_level_weight × second_level_score))`

`average_resource_score = avg(resource_score)`

P0 包级 `package_score` 与资源级平均分均由后端返回；前端只显示返回字段，不重新计算包级分、资源平均分、低分资源数或热力图得分。资源权重聚合、模态不适用指标归一化和 `not_applicable_reason` 可作为 P1 规则扩展。

| **场景** | **处理** |

- 分配结果必须保存 amount_source、contract_ratio、base_pool_amount、final_amount 和尾差说明。
- 锁定方案后所有关键配置只读。

### 收益分配公式

```text
contract_ratio_sum = data_provider_pool_ratio + sum(non_data_party_ratio_j)
non_data_contract_amount_j = total_revenue × non_data_party_ratio_j
data_provider_revenue_pool = total_revenue × data_provider_pool_ratio
data_provider_amount_i = data_provider_revenue_pool × md_dshap_normalized_weight_i
```

约束条件：

- `contract_ratio_sum = 1.000000`
- `data_provider_revenue_pool >= 0`
- `sum(md_dshap_normalized_weight_i) = 1`

# 8. MD-DShap 专题功能设计

## 8.1 产品定位

MD-DShap 是系统默认权重计算策略。基础 Shapley 仅在参与方数量较少时作为基线校验，不作为默认最终分配模式。MD-DShap 页面需要同时说明操作入口、算法参数和审计材料来源。

## 8.2 算法输入设计

| **输入项**       | **字段建议**                   | **来源模块**      | **校验规则**                                                         |
|------------------|--------------------------------|-------------------|----------------------------------------------------------------------|
| 数据源主体集合 P | participant_set                | 参与方管理        | 只允许 include_in_md_dshap=true 且 party_type=DATA_PROVIDER 的主体。 |
| 任务集合 T       | task_set                       | 数据资源/场景配置 | P0 可使用 default_task；多任务模式必须显式传入。                     |
| 效用函数 v(S,t)  | utility_function_snapshot_id   | 效用计算          | 必须有来源说明、公式说明和参数快照。                                 |
| 质量评分 q_i     | quality_factor / quality_score | 质量评估          | 取最新或用户选择的质量评估版本。                                     |
| 贡献度 c_i       | normalized_contribution        | 贡献度计算        | 合计应为 1。                                                         |
| 场景参数 s       | scenario_factor / scenario_id  | 系统参数/场景配置 | 系数必须大于 0。                                                     |
| 随机种子 seed    | seed                           | 算法参数          | 大于等于 0，默认 42。                                                |
| 采样轮次 n       | sample_rounds                  | 算法参数          | P1 完整开放；P0 可内置默认值。                                       |
| 收敛阈值 epsilon | epsilon                        | 算法参数          | 大于 0；P1 用于异步或长任务。                                        |

## 8.3 算法输出设计

| **输出项**                  | **字段建议**                           | **用途**                             |
|-----------------------------|----------------------------------------|--------------------------------------|
| participant_weight          | participant_weight / normalized_weight | 参与方综合权重，进入收益分配。       |
| task_level_weight           | task_level_weight_json                 | 多任务下每个任务维度的权重。         |
| marginal_contribution_trace | md_dshap_marginal_trace                | 审计算法边际贡献过程。               |
| approximation_note          | approximation_note                     | 披露采样、动态估值、近似假设和边界。 |
| algorithm_audit_snapshot    | algorithm_audit_snapshot_id            | 报告导出和审计追溯。                 |

## 8.4 v(S,t) 与效用函数规则

P0 默认规则：

`v(S,t) = sum(utility_value_i,t for i in S) + synergy_adjustment(S,t)`

其中 `synergy_adjustment(S,t)` P0 默认可为 0，P1 可扩展为多模态协同、数据互补性、场景协同项或专家调整项。空集合规则为 `v(∅,t)=0`。单参与方场景不启动完整 MD-DShap，直接输出权重 1，并在页面和报告中披露“单主体简化分配”。

## 12.1 必须记录日志的操作

| **操作类别** | **典型按钮**                                           | **必须记录内容**                                   |
|--------------|--------------------------------------------------------|----------------------------------------------------|

| 编辑         | 配置权重、编辑参与方、配置总收益、配置参数。           | 修改前值、修改后值、参数版本。                     |
| 删除/停用    | 删除数据包、停用参与方、停用约束。                     | 影响范围、二次确认、操作结果。                     |
| 计算         | 启动质量评估、数元计量、效用计算、MD-DShap、收益分配。 | 输入快照、参数快照、输出快照、算法版本、失败原因。 |
| 导出         | 导出报告、导出 CSV、导出日志。                         | report_id、文件名、格式、checksum、字段范围。      |
| 确认         | 锁定分配方案。                                         | 确认人、确认时间、人工说明、方案版本。             |

## 12.2 快照策略

1.  输入快照：上传 JSON、演示数据初始化、复制新版本时生成。
2.  参数快照：质量权重、数元系数、贡献因子、效用函数、MD-DShap 参数、收益池、合同比例方案在计算前生成。
3.  输出快照：每次计算成功后生成，导出报告引用输出快照。
4.  报告快照：每次导出报告时保存报告内容摘要、文件路径、checksum。
5.  历史快照不得修改；如需修订，生成新版本并保留来源版本号。

# 13. 数据库设计前置约束

1.  所有主业务表必须包含 `project_id` 或可通过外键追溯到 `project_id`。
2.  所有可重复执行的计算结果必须包含 `version_no`、`created_at`、`created_by`。
3.  金额字段使用 Decimal，不使用 float。
4.  权重字段统一保留 6 位，金额字段统一保留 2 位。
5.  历史结果引用的数据不得物理删除；删除类操作优先使用 status 标记。
6.  已导出报告引用的输入、参数、合同比例方案和结果必须不可变。
7.  大字段如输入快照、输出快照、边际贡献 trace 可使用 JSON 存储，但必须有 checksum。
8.  日志表应支持按项目、模块、对象、操作类型和时间范围查询。
9.  P1 用户权限表应支持菜单级、页面级、按钮级和导出级权限。
10. 报告文件路径和 checksum 必须入库，避免无法追溯导出版本。

# 14. UI 设计前置约束

1.  左侧导航必须采用六个一级菜单：系统首页、数据管理、数元贡献度计量、收益分配计算、报告生成与导出、系统管理；各二级功能必须与第 5.1 节一致。
2.  首页必须展示完整流程状态、项目总览、流程入口、风险提示和一键计算入口，避免用户不知道下一步做什么。
3.  每个模块顶部必须显示当前项目名称、状态、最近版本和风险提示。
4.  计算按钮必须先展示前置条件检查结果，未满足时禁用。
5.  表格操作列的高风险按钮必须二次确认。
6.  “重新计算”按钮必须提示不会覆盖历史结果。
7.  “导出”弹窗必须展示导出文件、字段范围、报告版本和免责声明。
8.  MD-DShap 页面必须突出“默认算法为 MD-DShap，基础 Shapley 仅作基线”。
9.  收益分配页面必须展示“总收益 -\> 合同比例划分 -\> 数据源收益池 -\> MD-DShap 权重分配 -\> 最终结果”的顺序。
10. 合同分配规则页必须展示 amount_source、contract_ratio、base_pool_amount 和最终金额来源，不能展示旧约束命中/未命中流程。
11. 审计页面必须支持从日志跳转到输入快照、参数快照和输出快照。
12. 已确认/已导出项目的编辑按钮必须置灰，并提供“复制新版本”入口。
13. P1 登录后，权限不足的按钮应隐藏或禁用，并显示角色原因。

1.  每个按钮必须至少验证：可见性、启用条件、权限、前置条件、成功输出、异常提示、日志记录。

# 18. 结论

本文档在 V1.4.3 需求规格说明书基础上，将系统需求拆成模块、页面路由、按钮行为、候选表、服务边界、导出字段、审计快照和验收映射。本次代码同步后，左侧导航固定为“系统首页、数据管理、数元贡献度计量、收益分配计算、报告生成与导出、系统管理”六个一级菜单，原有功能模块按当前页面归入对应二级功能。数据库设计可依据第 6、11、12、13 章区分已落 PostgreSQL DDL、JSON runtime 对象和正式库预留对象；UI 设计可依据第 5、7、8、14 章建立页面原型和交互流程；后端接口设计可依据第 10 章拆分服务和统一输入输出。

系统设计结论是：数据接入提供项目输入，质量评估、数元计量和效用计算形成算法输入；合同比例方案先将总收益划分为非数据主体合同金额和数据源收益池；MD-DShap 只生成数据源主体归一化权重；尾差处理、报告和审计日志负责结果追溯。系统结果仅为模拟参考，非法律结算。
