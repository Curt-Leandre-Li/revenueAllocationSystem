# 系统测试方案

## 1. Test Goal

The test plan verifies that the data revenue allocation system follows the frozen contracts, calculation boundaries, compliance boundaries, and UI workflow after coding starts.

## 2. Test Scope

- Input validation.
- Quality, metering, utility, DAUS, and Shapley calculation boundaries.
- Contract constraint application.
- Allocation reference output.
- Report artifact.
- UI workflow and state handling.
- Compliance wording for simulated data and medical sample boundaries.

## 3. Unit Tests

| Area | Planned Checks |
| --- | --- |
| Validation | Required fields, bad references, invalid ranges, missing simulated-data flag. |
| Quality | Stable score from fixed quality inputs. |
| Metering | Data Unit quantity and Token count handling. |
| Utility | Stable utility score from fixed inputs. |
| Contribution | DAUS and Shapley references remain intermediate signals. |
| Constraints | Minimum guarantee, cap, zero totals, confirmation flags. |

## 4. Integration Tests

- Valid sample package runs through all planned pipeline stages.
- Invalid sample package returns structured errors and stops calculation.
- Warning sample package preserves warnings into output and report.
- Output fields match `../api_and_data_contract.md`.

## 5. Contract Tests

- Input JSON sample matches required fields.
- Output JSON sample includes approved field names only.
- Error structure includes `code`, `message`, `path`, and `severity`.
- Report artifact includes simulated-data label and audit trace.

## 6. UI Acceptance Tests

- Pages follow the approved navigation flow.
- Validation errors disable pipeline run.
- Complete output enables report and export entry points.
- Shapley appears as contribution reference only.
- Simulated-data warning remains visible.

## 7. Report Acceptance Tests

- Report contains input summary, signal summary, constraints, allocation reference, warnings, and trace appendix.
- Report does not claim real hospital data.
- Report does not claim real business settlement.
- Report records confirmation-required flags.

## 8. Planned Validation Commands

The exact commands will be confirmed after code and package files exist. Current planned categories:

- unit test command;
- integration test command;
- contract sample validation command;
- UI smoke command if frontend is implemented;
- documentation whitespace check with `git diff --check`.

## 9. Phase 0 Boundary

This is a test plan before coding. It defines what must be tested later and does not provide execution evidence.
