# 数据库设计

## 1. Design Position

Current MVP does not depend on a real database. The planned implementation uses JSON/CSV files for sample input and in-memory computation for pipeline artifacts.

No database table has been created in Phase 0.

## 2. Storage Boundary

| Item | Current Phase 0 Decision | Notes |
| --- | --- | --- |
| Input samples | JSON/CSV files | Simulated data only. |
| Pipeline artifacts | In-memory objects during a run | May be exported as JSON. |
| Reports | File artifact after generation | Persistence path to be decided during coding. |
| User accounts | Not in MVP | No authentication or multi-user permission database. |
| Audit records | Report artifact and optional JSON trace | No external audit database. |

## 3. Logical Entity Design

Even without a real database, backend schema should be designed around these logical entities:

| Entity | Key | Main Fields |
| --- | --- | --- |
| InputPackage | `package_id` | scenario, participants, data_units, contract_constraints |
| Participant | `participant_id` | display_name, role |
| DataUnit | `data_unit_id` | participant_id, token_count, quantity, quality_inputs, utility_inputs |
| ContractConstraints | package scope | currency, revenue_pool, minimum_guarantees, caps, confirmation flags |
| ValidationRecord | package scope | errors, warnings |
| QualityRecord | `data_unit_id` | completeness, validity, consistency, traceability, quality_score |
| MeteringRecord | `data_unit_id` | token_count, quantity, metered_units |
| UtilityRecord | `data_unit_id` | business_relevance, reuse_potential, utility_score |
| ContributionRecord | `participant_id` | base_signal, daus_score, shapley_weight, contribution_reference |
| AllocationRecord | `participant_id` | base_amount, constraints, final_allocation_reference, warnings |
| ReportRecord | `report_id` | title, sections, warnings, audit_trace |

## 4. Relationship Design

```text
InputPackage 1 -> many Participant
InputPackage 1 -> many DataUnit
Participant 1 -> many DataUnit
DataUnit 1 -> one QualityRecord
DataUnit 1 -> one MeteringRecord
DataUnit 1 -> one UtilityRecord
Participant 1 -> one ContributionRecord
Participant 1 -> one AllocationRecord
InputPackage 1 -> one ReportRecord
```

## 5. Field Naming Rules

- JSON fields use snake_case.
- IDs use stable string values.
- Use `data_unit_id` for Data Unit identity.
- Use `token_count` only for Token quantity.
- Use `final_allocation_reference` for the final reference amount.
- Do not introduce Effective DU.

## 6. Future Persistence Option

If later implementation needs persistence, the design may be mapped to relational tables or document storage after explicit approval. That future decision must not be treated as part of the current MVP.

## 7. Phase 0 Conclusion

The database design is frozen as a logical data model. The current MVP does not create real database tables and does not require a database service.
