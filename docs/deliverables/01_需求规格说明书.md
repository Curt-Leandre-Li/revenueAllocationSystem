# 需求规格说明书

## 1. System Positioning

The system is a generic data revenue allocation system for software-copyright demonstration, implementation planning, and audit-oriented reporting. It helps users input simulated data packages, inspect contribution signals, calculate allocation references, and generate a report.

The system does not produce legal settlement instructions. The final allocation reference may be affected by contract terms, minimum guarantee, cap, expert confirmation, and human confirmation.

## 2. User Roles

| Role | Responsibility | Main Needs |
| --- | --- | --- |
| Business reviewer | Reviews simulated revenue allocation scenarios. | Understand input, contribution signals, and final allocation reference. |
| Data provider representative | Reviews participant-related Data Unit contribution. | Confirm Data Unit ownership and contribution reference. |
| Algorithm reviewer | Reviews formulas and intermediate signals. | Inspect quality, utility, DAUS, and Shapley calculation layer. |
| Compliance reviewer | Reviews risk wording and audit trace. | Confirm simulated-data labels and avoid real business claims. |
| System operator | Runs the local demo and archives outputs. | Prepare input, run pipeline, export report artifacts. |

## 3. Usage Scenarios

- Load or create a simulated data package.
- Validate participants, Data Units, Token counts, quality inputs, utility inputs, and contract constraints.
- Run the deterministic pipeline.
- Inspect intermediate quality, metering, utility, DAUS, and Shapley contribution references.
- Apply contract constraints to generate a final allocation reference.
- Generate an audit-readable report with simulated-data labels.

## 4. Functional Requirements

| ID | Requirement | Acceptance |
| --- | --- | --- |
| FR-01 | The system shall accept a simulated input package. | Required fields are validated and errors include field paths. |
| FR-02 | The system shall calculate quality signals for each Data Unit. | Output includes quality dimensions and a quality score. |
| FR-03 | The system shall calculate Data Unit metering results. | Output preserves Data Unit IDs and Token quantity where provided. |
| FR-04 | The system shall calculate utility signals. | Output includes utility dimensions and utility score. |
| FR-05 | The system shall produce DAUS contribution signals. | DAUS is labeled as an intermediate signal. |
| FR-06 | The system shall use Shapley as a computation-layer weight method. | Shapley output is not labeled as final revenue allocation. |
| FR-07 | The system shall apply contract constraints. | Minimum guarantee, cap, and confirmation flags are reflected in output. |
| FR-08 | The system shall produce an allocation reference. | Output uses `final_allocation_reference`, not settlement wording. |
| FR-09 | The system shall generate an audit report artifact. | Report includes inputs, signals, constraints, warnings, and trace records. |
| FR-10 | The system shall support a lightweight demo UI after coding. | UI displays approved API fields and does not invent calculation results. |

## 5. Input Requirements

- `package_id` must identify the input package.
- `scenario.is_simulated` must be true for sample data.
- Participants must have stable `participant_id` values.
- Data Units must have stable `data_unit_id` values and participant references.
- Token quantity may be recorded as `token_count`.
- Contract constraints may include revenue pool, minimum guarantees, caps, and confirmation flags.

## 6. Output Requirements

- Structured validation result.
- Quality result list.
- Metering result list.
- Utility result list.
- Contribution result list with DAUS and Shapley contribution references.
- Allocation reference with constraint adjustments and confirmation flags.
- Report artifact with simulated-data warning and audit trace.

## 7. Page Requirements

- Input Package.
- Validation Results.
- Pipeline Review.
- Signal Details.
- Contribution And Allocation.
- Audit Report.
- Export.

## 8. Report Requirements

The report shall include:

- Input summary.
- Formula and assumption summary.
- Intermediate signal tables.
- Contract constraint table.
- Allocation reference summary.
- Warning and confirmation-required section.
- Audit trace appendix.

## 9. Non-Functional Requirements

- Deterministic output for identical input.
- No hidden medical assumption in generic modules.
- No new production dependency without explicit approval.
- JSON field names use snake_case.
- Simulated data labels must be visible in output and report.

## 10. Exclusions

- No MAR.
- No Effective DU.
- No `token_weighting` main process.
- No real payment, settlement, or legal conclusion.
- No real hospital data claim.
- No production authentication or multi-user permission system in MVP.
