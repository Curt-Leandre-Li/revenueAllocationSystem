# 系统详细设计（功能设计）

## 1. Design Goal

The system is designed as a generic data revenue allocation workflow. It receives a simulated input package, validates it, calculates intermediate signals, applies contract constraints, and generates an allocation reference plus audit report.

## 2. Functional Modules

| Module | Responsibility | Input | Output |
| --- | --- | --- | --- |
| Input Package Module | Load and normalize simulated input. | JSON/CSV sample data. | `InputPackage`. |
| Validation Module | Validate required fields, references, ranges, and simulated-data flag. | `InputPackage`. | `ValidationResult`. |
| Quality Module | Evaluate completeness, validity, consistency, and traceability. | Valid Data Units. | `QualityResult[]`. |
| Metering Module | Record Token quantity and Data Unit quantity. | Valid Data Units. | `MeteringResult[]`. |
| Utility Module | Calculate business relevance and reuse potential signals. | Valid Data Units. | `UtilityResult[]`. |
| Contribution Module | Combine intermediate signals into DAUS and Shapley contribution references. | Quality, metering, utility. | `ContributionResult[]`. |
| Constraint Module | Apply revenue pool, minimum guarantee, cap, and confirmation flags. | Contribution references and contract constraints. | `AllocationReference`. |
| Report Module | Generate audit-readable report artifact. | Pipeline artifacts. | `ReportArtifact`. |
| Demo UI Module | Display workflow and artifacts after frontend coding. | API output contract. | User-visible pages. |

## 3. Pipeline Flow

```text
InputPackage
-> ValidationResult
-> QualityResult[]
-> MeteringResult[]
-> UtilityResult[]
-> ContributionResult[]
-> AllocationReference
-> ReportArtifact
```

Validation errors stop the calculation. Warnings may continue into output when the input is usable but needs review.

## 4. Responsibility Boundaries

- Validation decides whether the package can continue.
- Quality, metering, and utility modules produce intermediate signals only.
- DAUS is an intermediate contribution signal.
- Shapley is a computation-layer weight method.
- Contract constraints convert contribution references into a final allocation reference.
- Human or expert confirmation remains outside automatic calculation.

## 6. Error Handling

Errors use:

- `code`
- `message`
- `path`
- `severity`

Allowed severities are `error` and `warning`.

## 7. Audit Records

Each run should preserve:

- input package ID;
- validation summary;
- formula version or documented formula note;
- generated artifacts;
- simulated-data label;
- confirmation-required flags;
- report generated time.
